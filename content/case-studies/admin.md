## Context

As MathTrack grew, so did the operational work required to support students. Every new program, enrollment requirement, transcript request, and reporting obligation introduced another manual process. Without an interface for staff to use, the work either stayed in spreadsheets or came to me as a request to hard-code a change into the application.

Our administrative staff covered a wide range of responsibilities, from reviewing applications and enrolling students to moving students between cohorts, generating transcripts, and preparing data for state reporting. Information was spread across multiple systems, with HubSpot handling some tasks while others required direct changes to the application or database.

There was no upfront plan to build a complete administrative platform. I built admin interfaces alongside new features as operational needs appeared, so staff could configure, monitor, and fix the things each feature introduced without waiting for a code change.

Most day-to-day administrative work now happens without me making manual changes in code. Staff can manage student records, process enrollments, organize cohorts, generate transcripts, and handle many tasks that previously meant a spreadsheet workaround or a hard-coded fix. A small team has been able to absorb organizational growth without proportionally increasing administrative overhead.

## Admin With Each Feature

Most administrative tooling was built when a new student-facing feature created operational work that had to be handled repeatedly. Enrollment flows needed a way to place students in courses and cohorts. Grading needed a queue for instructors and staff to review submissions. Departments, programs, and course configuration each needed their own management screens as those parts of the platform came online.

In practice, shipping a feature usually meant shipping the admin interface for it at the same time. The student-facing side solved the learner experience; the admin side solved whatever our team needed to manage after launch.

## What Accumulated Over Time

The result is a set of admin tools shaped by the features they support rather than a single unified system designed in advance. Some interfaces are closely related because they manage adjacent parts of the platform, but they were not built to share a common framework or follow one interaction model across the board.

That approach matched how the organization actually grew. Each tool addressed a specific operational problem when it appeared, and the collection of those tools is what staff use to run the platform today.
